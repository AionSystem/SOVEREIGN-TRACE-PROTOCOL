## WHY THREE LICENSES?

Most software ships with one license. This repository ships with three.
That is intentional — each license closes a different gap that a single
license leaves open.

Here is what each one does in plain language, and which one applies to you.

---

## LICENSE 1 — APACHE 2.0 (The Open License)

**File:** `LICENSE`
**Who it applies to:** Individual users, researchers, students, non-commercial
open source projects, internal enterprise tools that don't offer certification
to others.

**What you can do:**
- Use the stamp function in your own projects — free, forever
- Modify the code for your own use
- Distribute your version to others
- Build products on top of it

**What you must do:**
- Keep the copyright notice and attribution to Sheldon K. Salmon
- Include a copy of the Apache 2.0 license with any distribution
- Note what you changed if you modified the code
- Include the `NOTICE` file with any distribution (Apache 2.0 requires this)

**The patent clause — important:**
Apache 2.0 includes a patent retaliation clause. If you use this software
and then sue Sheldon K. Salmon (or any contributor) for patent infringement
related to this project — your Apache 2.0 patent license terminates
automatically the moment you file. You lose the right to use the software
and face potential infringement liability simultaneously.

This is standard Apache 2.0 language. It is not aggressive — it is
defensive. It prevents the scenario where someone uses open source software,
patents a concept in it, and then sues the original author.

## LICENSE 2 — GPL v3 (The Copyleft License)

**File:** `LICENSE-GPLv3` (see gnu.org/licenses/gpl-3.0 for full text)
**Who it applies to:** Anyone who modifies this code and **distributes**
the modified version publicly — including as part of a product or service.

**What this means in plain language:**
If you fork this repository, change the code, and distribute your version
to other people or organizations — your modified version must also be
open sourced under GPL v3. You cannot take this code, improve it,
and lock those improvements in a proprietary product.

**Important — internal modifications are NOT covered:**
If you modify the code for internal use and do **not** distribute it to
third parties, the GPL v3 copyleft does **not** require you to release
your modifications. You can keep them private.

**What you can do:**
- Modify the code freely for internal use (no distribution = no GPL trigger)
- Use it in open source projects that are also GPL-compatible

**What you cannot do:**
- Distribute a modified version of this code in a proprietary product
  without open-sourcing your modifications under GPL v3

**Important — network interaction (SaaS) does NOT trigger GPL v3:**
The GPL v3 copyleft applies only when you **distribute** copies of the
software (including modified versions). Running the software on a server
(SaaS) without distributing copies does **not** require you to release
your modifications under GPL v3. However, if you offer certification or
commercial stamping services using this protocol, the Commercial License
may still be required regardless of distribution.

**Why this exists:**
The GPL v3 closes the corporate fork loophole. Without it, a company could
take this stamp function, improve the Hebrew calendar algorithm, build it
into an enterprise product, and lock those improvements away. The GPL v3
ensures that improvements to this public infrastructure stay public.

This has been tested in court. *Jacobsen v. Katzer* (Federal Circuit, 2008)
confirmed that open source license conditions are enforceable as copyright
conditions — not just contractual promises. GPL v3 violations are copyright
infringement, actionable under U.S. law for statutory damages up to $150,000
per willful violation (17 U.S.C. § 504). Other jurisdictions may have different
remedies. Consult local counsel.
